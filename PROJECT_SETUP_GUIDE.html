PRD (기획서) 작성하기

PRD란?

PRD(Product Requirements Document)는 "우리가 뭘 만들 건지"를 문서로 정리한 것입니다. 코딩 전에 팀 전체가 같은 그림을 보고 시작하기 위한 설계도입니다.

😵
PRD 없이 시작하면
"로그인은 누가 만들지?"
"DB는 뭘 쓰기로 했지?"
"이 기능 필요했나?"
→ 중간에 방향이 바뀌고 코드 버림
🎯
PRD 있으면
전원이 같은 목표를 공유
기능 범위가 명확
TASK.md로 바로 변환 가능
→ 삽질 최소화

PRD 템플릿

아래 구조를 그대로 따라 작성하면 됩니다.

PRD.md
📄 PRD 템플릿
1. 프로젝트 개요
프로젝트 이름: (예: AI Study Buddy)
한 줄 설명: (예: 학습 자료를 업로드하면 AI가 퀴즈를 만들어주는 웹앱)
목표: 이 프로젝트가 해결하려는 문제
기간: (예: 3주)
팀 규모: (예: 7명)
2. 타겟 사용자
주요 사용자: 누가 쓸 것인가? (예: 대학생, 직장인)
사용 시나리오: 사용자가 어떤 흐름으로 사용하는지 구체적으로
3. 핵심 기능 목록
각 기능을 우선순위와 함께 나열

P0 필수사용자 로그인/회원가입v1.0 P0 필수파일 업로드 + AI 퀴즈 생성v1.0 P0 필수퀴즈 풀기 + 채점v1.0 P1 중요학습 기록 대시보드v1.1 P2 있으면 좋음소셜 공유, 랭킹v2.0
4. 기술 스택
프론트엔드: (예: React, Next.js, HTML/JS)
백엔드: (예: FastAPI, Flask, Express)
AI: (예: OpenAI API, LangChain, HuggingFace)
DB: (예: PostgreSQL, MongoDB, SQLite, Supabase)
배포: (예: Vercel, AWS, Docker)
5. 화면(페이지) 목록
각 화면이 무슨 역할인지, 어떤 API를 호출하는지

• 로그인/회원가입 페이지 → POST /auth/login, POST /auth/register
• 메인 대시보드 → GET /quizzes
• 파일 업로드 페이지 → POST /upload
• 퀴즈 풀기 페이지 → GET /quiz/:id, POST /quiz/:id/submit
• 결과 페이지 → GET /quiz/:id/result
6. API 엔드포인트 설계
POST /auth/register — 회원가입
POST /auth/login — 로그인
POST /upload — 파일 업로드
POST /generate-quiz — AI 퀴즈 생성
GET  /quiz/:id — 퀴즈 가져오기
POST /quiz/:id/submit — 답안 제출 + 채점
7. 데이터 모델 (DB 구조)
User: id, email, password_hash, created_at
Quiz: id, user_id, title, questions_json, created_at
Submission: id, quiz_id, user_id, answers_json, score
💡
Claude Code 활용 팁: 아이디어만 있으면 Claude Code에게 "이런 서비스를 만들 건데 PRD 작성해줘"라고 하면 위 형식으로 초안을 만들어줍니다. 팀원들이 검토하고 수정하면 됩니다.

PRD → 프로젝트 구조 설계

핵심 원칙: "기능별로 파일을 분리"

PRD에서 정의한 기능들이 각각 별도의 파일/폴더가 되도록 설계합니다.
이렇게 해야 여러 명이 동시에 작업해도 충돌이 나지 않습니다.

PRD의 기능 → 폴더 구조로 변환
PRD 기능 목록
📌 인증 (로그인/회원가입)
📌 파일 업로드
📌 AI 퀴즈 생성
📌 퀴즈 풀기/채점
📌 프론트엔드 UI
→
project/ ├── app.py # 진입점 (import만) ├── routes/ │ ├── auth.py # ← 인증 │ ├── upload.py # ← 업로드 │ └── quiz.py # ← 퀴즈 ├── services/ │ ├── ai_service.py # ← AI 생성 │ ├── quiz_service.py # ← 채점 │ └── file_service.py # ← 파일 처리 ├── models/ │ ├── user.py # ← User 모델 │ └── quiz.py # ← Quiz 모델 └── frontend/ ├── login.html # ← 인증 UI ├── upload.html # ← 업로드 UI └── quiz.html # ← 퀴즈 UI

폴더 구조 설계 규칙

폴더역할규칙
routes/API 엔드포인트PRD의 API 하나당 파일 하나
services/비즈니스 로직PRD의 핵심 기능 하나당 파일 하나
models/DB 모델PRD의 데이터 모델 하나당 파일 하나
frontend/화면 (UI)PRD의 화면 하나당 폴더/파일 하나
tests/테스트 코드위 파일과 1:1 매칭
config/설정환경별 설정 파일
⚠️
핵심: app.py (진입점)에는 import와 서버 실행 코드만 넣습니다. 실제 로직은 전부 routes/, services/에 분리합니다. 이래야 app.py를 건드리는 사람이 최소화됩니다.

TASK.md 작성하기

TASK.md란?

프로젝트의 모든 작업 목록을 Phase별로 정리한 문서입니다. 각 Task에 담당자, 수정할 파일, 브랜치명, 의존성을 명시합니다.

TASK.md 작성 3단계

1
PRD의 기능을 작업 단위로 쪼갠다
하나의 기능을 1~2일 안에 완료할 수 있는 크기로 분해
2
각 Task가 건드리는 파일을 명시한다
같은 Phase 내에서 파일이 겹치지 않도록 배치
3
Phase로 묶어 순서를 정한다
의존성이 없는 작업끼리 같은 Phase → 동시 진행 가능

Phase 분류 기준

Phase성격동시 진행예시
Phase 0프로젝트 세팅팀장 혼자저장소 생성, 초기 구조, 인터페이스 정의
Phase 1독립 모듈 개발전원 동시서로 다른 파일을 각자 개발
Phase 2모듈 연결대부분 동시Phase 1 결과물을 조립
Phase 3공통 파일 조립한 명씩 직렬app.py에 라우트 등록 등
Phase 4통합 + 배포전원 참여테스트, 버그 수정, 배포

TASK.md 템플릿

TASK.md
Phase 0 프로젝트 세팅 팀장만 
# Phase 0: 프로젝트 세팅 (팀장)

- [ ] GitHub 저장소 생성 + 팀원 초대
- [ ] 초기 폴더 구조 생성
- [ ] .gitignore, .env.example, requirements.txt
- [ ] 인터페이스 정의 (interfaces.py)
- [ ] develop 브랜치 생성 + 보호 규칙
- [ ] 이 TASK.md를 저장소에 커밋
Phase 1 독립 모듈 개발 동시 진행 가능 ✅ 
# Phase 1: 독립 모듈 개발 (동시 진행 가능)

- [ ] 담당: OOO  Task 제목
      파일: 수정할 파일 목록
      브랜치: feature/xxx

- [ ] 담당: OOO  Task 제목
      파일: 수정할 파일 목록
      브랜치: feature/yyy

# ⚠️ 같은 Phase 내에서 파일이 겹치지 않는지 확인!
Phase 2 모듈 연결 Phase 1 머지 후 
# Phase 2: 모듈 연결 (Phase 1 완료 후)

- [ ] 담당: OOO  Task 제목
      파일: 수정할 파일 목록
      의존: Phase 1의 어떤 Task가 필요한지
      브랜치: feature/zzz
Phase 3 공통 파일 조립 + 통합 한 명씩 직렬! 
# Phase 3: 공통 파일 조립 (한 명씩 순서대로!)

- [ ] 담당: OOO  app.py에 라우트 등록
      파일: app.py
      순서: 1번째

- [ ] 담당: OOO  app.py에 미들웨어 등록
      파일: app.py
      순서: 2번째 (위 Task 머지 후)

# Phase 4: 통합 테스트 + 배포
- [ ] 전원 통합 테스트
- [ ] develop → main PR + 릴리스
💡
파일 겹침 체크 방법: Phase 1의 모든 Task에서 "파일" 항목을 모아보세요. 같은 파일이 두 번 이상 나오면 → 둘 중 하나를 Phase 2로 옮기거나, 파일을 더 쪼개야 합니다.

팀원 역할 분배

분배 기준

고려 사항설명
경험/관심사프론트 해본 사람은 프론트, AI 관심 있는 사람은 AI 서비스
작업량 균등한 사람에게 몰리지 않도록. Phase 1에서 1인 1~2 Task
병목 최소화다른 사람이 기다려야 하는 작업(DB 모델 등)은 빠른 사람에게
리뷰 교차내가 만든 코드는 다른 사람이 리뷰. 서로 코드를 이해하게 됨

7인 팀 역할 분배 예시

👑 팀장 (1명)
프로젝트 세팅
PRD 총괄
공통 파일 관리
PR 리뷰 + 머지 관리
🔧 백엔드 (2명)
API 라우트 개발
DB 모델 설계
서비스 로직 구현
🤖 AI (2명)
AI API 연동
프롬프트 엔지니어링

🎨 프론트엔드 (2명)

📌
팀장도 개발합니다! 프로젝트 세팅 후에는 백엔드나 AI 쪽 작업을 함께 합니다. 별도 관리만 하는 역할이 아닙니다.

분배 후 TASK.md에 반영

역할이 정해지면 TASK.md의 각 Task에 담당자를 배정합니다.

# Phase 1 — 담당자 배정 완료

- [ ] 담당: 민수(백엔드)  DB 모델 정의
      파일: models/user.py, models/quiz.py
      브랜치: feature/db-models

- [ ] 담당: 지연(AI)  OpenAI 퀴즈 생성 서비스
      파일: services/ai_service.py
      브랜치: feature/ai-quiz-gen

- [ ] 담당: 현우(프론트)  로그인/회원가입 페이지
      파일: frontend/pages/login.html, frontend/js/auth.js
      브랜치: feature/auth-ui

# ✅ 파일 겹침 없음 → 동시 작업 가능!
5

Claude Code로 실행하기

각 단계에서 Claude Code 활용법

Step 1 PRD 작성
팀장이 Claude Code에게 요청:
# Claude Code 터미널에서

"우리 팀은 7명이고, [서비스 설명]을 만들려고 해.
PRD를 작성해줘. 핵심 기능, 기술 스택, API 설계,
DB 모델까지 포함해서 PRD.md로 저장해줘."
💡
Claude가 PRD 초안을 만들면, 팀원들이 검토하고 의견을 반영해서 수정합니다.
Step 2 프로젝트 구조 생성
PRD 확정 후 Claude Code에게 요청:
"PRD.md를 기반으로 프로젝트 초기 구조를 만들어줘.
7명이 동시 작업할 수 있도록 기능별로 파일을 분리해줘.
각 파일에 인터페이스(함수 시그니처)만 정의해줘.
실제 구현은 pass로 비워둬."
Step 3 TASK.md 생성 
"PRD.md와 현재 프로젝트 구조를 보고
TASK.md를 만들어줘. Phase별로 나누고,
각 Task에 담당할 파일과 브랜치명을 포함해줘.
같은 Phase 내에서 파일이 겹치지 않게 해줘."
Step 4 각 팀원의 개발 시작
각 팀원이 자기 Task를 Claude Code로 개발:
# 팀원 각자 자기 컴퓨터에서
git checkout develop
git pull origin develop
git checkout -b feature/내-작업-브랜치

# Claude Code 실행
"TASK.md에서 내 담당인 [Task 이름]을 구현해줘.
인터페이스는 이미 정의되어 있어."

# 완료 후
git add 변경된파일들
git commit -m "feat: Task 설명"
git push origin feature/내-작업-브랜치
# → GitHub에서 PR 생성

Claude Code에 프로젝트 컨텍스트 주기: CLAUDE.md

프로젝트 루트에 CLAUDE.md 파일을 만들면, Claude Code가 항상 이 내용을 인식합니다.

# CLAUDE.md 예시

# 프로젝트: AI Study Buddy
7인 팀 프로젝트. OpenAI API 기반 학습 퀴즈 생성 웹앱.

## 기술 스택
- Backend: FastAPI + Python 3.11
- Frontend: HTML/JS (Vanilla)
- AI: OpenAI API (gpt-4o-mini)
- DB: SQLite + SQLAlchemy

## 컨벤션
- 커밋 메시지: feat/fix/docs/refactor/test: 설명
- 브랜치: feature/기능명, fix/버그명
- 한국어 주석 OK, 변수명은 영어

## 구조
- routes/ : API 라우트 (기능별 파일 분리)
- services/ : 비즈니스 로직
- models/ : DB 모델
- frontend/ : 프론트엔드 페이지

## 주의
- .env 파일 절대 커밋 금지
- app.py 수정 시 팀 채팅방에 선언 후 작업
💡
CLAUDE.md는 팀원 전원이 공유합니다. 저장소에 커밋하면 모든 팀원의 Claude Code가 동일한 컨텍스트를 가지게 됩니다. 코딩 스타일, 프로젝트 규칙을 한 곳에서 관리하는 셈입니다.

실전 예시: AI 챗봇 프로젝트
